SpringBoot 小补充记录

  • 官方文档
  • 在没有 SpringBoot 前要手动配置很多的依赖，还要写各种配置文件，比较麻烦
  • 有了 SpringBoot 后很多东西都自动配置好了，只需要会用就可以了，既方便了开发也方便了部署
  • 如果需要配置，SpringBoot 也可以手动进行自定义配置，即在 application.properties 文件或者 application.yaml 文件中配置就好了

创建一个 SpringBoot 项目记录

  • 打开 spring initializr 创建并下载项目，再用一个开发工具打开

  • 也可以用集成开发工具进行创建

  • 创建完成打开后就可以写代码了，比如创建的是 web 项目，就在与 main 函数同一个包下写 Controller 和 Service 和 DAO 等等

  • 启动项目，访问 http://localhost:8080/hello 返回一串文本——Hello Spring Boot !

  • 打包项目，以前要打成一个 war 包，现在直接用 maven 进行打包，即运行 package 命令就可以了

    • 打包完成后可以在 target 目录下看到有一个 jar 包，直接用 java -jar xxx.jar 运行就可以了

  • 在运行 jar 包时也可以指定配置，比如：java -jar xxx.jar --server.port=1234 指定服务器端口为 1234

  • 当然也可以指定配置文件

    • 官方文档

    • 创建配置文件，比如在和 jar 包同一目录下：

    • 启动时指定配置文件：java -jar xxx.jar --spring.config.location=./application.properties

SpringBoot 场景启动器 starter 记录

  • 类似于 spring-boot-starter-xxx 的依赖就是场景启动器，作用是引入当前场景的所有依赖

  • Spring 的场景：https://docs.spring.io/spring-boot/reference/using/build-systems.html

  • 在 SpringBoot 中，要用什么功能，只需要导入对应的场景启动器就可以了

    • 比如：spring-boot-starter-web

  • 场景有什么用？每一个场景启动器把场景要用的 jar 包全部引用好了

    • 比如在 spring-boot-starter-web 场景中有下面的依赖
      • spring-boot-starter
      • spring-boot-starter-json
      • spring-boot-starter-tomcat
      • spring-web
      • spring-webmvc
      • ...
    • 而在其中像 spring-boot-starter-tomcat 依赖也是一个场景启动器，点进去可以看到有很多引用的依赖：
      • jakarta.annotation-api
      • tomcat-embed-core
      • tomcat-embed-el
      • tomcat-embed-websocket
      • ...

  • 推荐的场景启动器命名

    • 官方的：spring-boot-starter-xxx
    • 第三方：xxx-spring-boot-starter

  • 为什么在 SpringBoot 中的依赖没有写版本号：因为在父项目 spring-boot-starter-parent 的父项目 spring-boot-dependencies 中进行约束了

  • 但是如果是 Spring 没有约束的依赖，要指定依赖版本号

  • 如果不想用默认的版本

    • 手动使用 <version> 标签指定版本号
    • 或者在 properties 标签中指定版本号 <xxx.version></xxx.version>

  • 建议命名的官方文档

SpringBoot 自动配置记录

自动配置的基本情况记录

  • SpringBoot 遵循 “约定大于配置” 的策略，即导入场景后大部分东西已经配置好了，比如数据库的数据源就可以在 application.properties 文件中配置
  • 如果没有手动配置的情况下就会使用默认的配置，即默认已经有配置好了
  • 但是有些属性是没有默认值的，比如数据库的用户和密码要手动指定
  • 如何查看一个属性有没有默认值，在属性配置文件中用 Ctrl 键加鼠标左键单击即可查看
  • SpringBoot官方所有的默认配置

自动配置的部分原理记录

场景启动器导入组件部分记录

  • 场景启动器的配置流程是怎么做的

    • 创建项目，选中场景并创建

    • 此时导入了场景启动器，场景启动器做了两件事

      • 导入场景所需依赖
      • 导入基础依赖 spring-boot-starter

    • 进入项目启动流程

    • 被注解 @SpringBootApplication 标注的主程序启动，点进注解中，有三个重点的注解

      • @SpringBootConfiguration 是容器中的配置类组件
      • @EnableAutoConfiguration 是开启自动配置
      • @ComponentScan 是包扫描规则

    • 其中，注解 @EnableAutoConfiguration 最重要，里面有一个注解 @Import(AutoConfigurationImportSelector.class) 导入的是一个特殊的组件，点进去可以用打断点的方式进行分析

      • 断点打好后，启动程序，下一步后可以看到调用了 getCandidateConfigurations() 方法并返回一个 List ，点开这个 list 可以看到有很多个配置名，都是以 xxxAutoConfiguration 为命名，就是这一步导入了自动配置类

      • 开始调试 Step Into 进入这个方法，发现是调用了本类中的另一个方法，使用了 ImportCandidates 类中的 load 方法进行导入，再 Step Into 进入 load 方法

      • 进入到了 ImportCandidates 类中的 load 方法，可以看到又调用了同一类中的 findUrlsInClasspath 方法进行查找导入，Step Into 进去，可以发现 location 的值为 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports

      • 由此可知 SpringBoot 有在 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 下导入自动配置

    • 可以打开一个自动配置类分析一下：DataSourceAutoConfiguration

      • 可以发现其是通过注解 @Conditional 和注解 @ConditionalOnMissingBean 进行判断从而决定是否把组件添加到容器中的
      • 如果不想用自动配置好的则可以自己根据上面的两个注解条件判断添加相应的组件

  • 总结为：

    • 导入场景启动器
    • 场景启动器会在 META-INF/xxx/imports 下导入自动配置类（xxxAutoConfiguration）
    • 在自动配置类中会使用注解 @Conditional 和注解 @ConditionalOnMissingBean 导入组件到容器中

自动配置类分析记录

  • 以一个自动配置类为例进行分析：DataSourceAutoConfiguration

  • 可以发现代码 @EnableConfigurationProperties(DataSourceProperties.class) 即是开启进行配置文件与配置类属性的绑定，并把配置类 xxxProperties 放到容器中，即配置绑定

    • 注解 @EnableConfigurationProperties 中的值即是配置类，点击进入配置类 DataSourceProperties
    • 有一个注解 @ConfigurationProperties(prefix = "spring.datasource") 其中的 prefix 属性即是在配置文件 application.properties 中的属性配置时的前缀
    • 当自动配置类要添加一些组件到容器中的时候，需要的属性从配置类中获取，而配置类的属性从配置文件 application.properties 中获取

总结：在导入场景启动器时从 xxx-autoconfigure.jar 包下的路径 META-INF 下导入自动配置类（xxxAutoConfiguration），然后在自动配置类中又有添加组件，并把配置文件和属性类进行属性的绑定，把属性类放入容器中，可以用注解 @Enablexxx 开启对应的功能，比如： @EnableAsync ，简畧的流程图如下：

SpringBoot 日志记录

  • 在项目开发时，难免会用到日志而不是控制台输出
  • SpringBoot 默认使用的组合是 slf4j + logback 作为日志记录（由场景启动器 spring-boot-starter-logging 知），官方文档

日志门面和日志实现记录

  • 日志分为：日志门面（Logging Facade）和日志实现（Logging Implementation），下面是 AI 生成的介绍：
    • 常见日志门面：
      • SLF4J (Simple Logging Facade for Java)：目前主流的日志门面，由 Log4j/Logback 的作者开发；支持与多种日志实现（Logback、Log4j 2、JUL等）集成；通过适配器（如 slf4j-log4j、slf4j-jdk）桥接到其他日志框架。
      • Apache Commons Logging (JCL)：早期流行的日志门面，设计较为简单，动态绑定日志实现；因类加载问题逐渐被 SLF4J 取代。
      • JBoss Logging：主要用于 JBoss/WildFly 生态，支持国际化和消息格式化。
    • 常见日志实现：
      • Log4j 2：Apache 开发的下一代日志框架，性能优异（支持异步日志）；提供丰富的插件和扩展机制，支持 JSON/YAML 配置；兼容 SLF4J 可以通过 log4j-slf4j-impl 适配器与 SLF4J 集成。
      • Logback：SLF4J 的原生实现，由 Log4j 1.x 作者开发，性能优于 Log4j 1.x；支持自动重新加载配置、细粒度过滤等特性；通常与 SLF4J 直接搭配使用。
      • Log4j 1.x：旧版日志框架，已停止维护，不推荐新项目使用。
      • Java Util Logging (JUL)：JDK 自带的日志实现，功能简单，扩展性较弱；可通过 slf4j-jdk 适配器桥接到 SLF4J。
      • Tinylog：轻量级日志框架，无外部依赖，适合小型项目；支持 SLF4J 接口。
    • 典型搭配方案：
      • SLF4J + Logback：最经典的组合，Logback 作为 SLF4J 的默认实现，不需要额外适配。
      • SLF4J + Log4j 2：高性能组合，适合需要异步日志或复杂配置的场景。
      • SLF4J + JUL：适用于依赖 JDK 内置日志的简单场景。

日志级别记录

  • 日志的级别：
    • SLF4J 支持的日志级别排序从低到高分别为：TRACE < DEBUG < INFO < WARN < ERROR
    • Logback 日志实现多了两个级别，排序为：ALL < TRACE < DEBUG < INFO < WARN < ERROR < OFF

SpringBoot 环境隔离记录

  • 在初学习 Spring 的时候，有接触过一个注解，即 @Profile 注解，这个注解可以配合 SpringBoot 的环境隔离机制使用

注解 @Profile 记录

  • 本质是一个 @Conditional 注解

  • 根据不同的环境，启用不同环境相对应的组件，例如：数据源的注入，开发时用一个数据源，发布时用另一个数据源，此时就可以用到 @Profile 注解了

  • 如何指定环境：在配置文件 application.properties 中激活相应的环境，例如激活 myenv 环境

  • 无论有没有在配置文件中指定环境，都会有一个名为 default 的环境，即默认环境

环境隔离步骤记录

  • 第一步：定义环境：在配置文件 application.properties 中定义环境

  • 第二步：定义这个环境下生效哪些组件或者哪些配置，官方文档

    • 生效组件用 @Profile 注解
    • 生效配置用 application-{环境标识}.properties 文件进行配置

  • 第三步：激活特定环境，官方文档

    • 在 application.properties 中使用 spring.profiles.active=[环境标识数组] 激活环境
    • 也可以在启动项目中添加参数进行激活 java -jar demo.jar --spring.profiles.active=[环境标识数组]
    • 也可以在 application.properties 中使用 spring.profiles.include=[环境标识数组] 激活环境

  • 当配置好环境后，生效的配置有：

    • 默认配置
    • 使用 spring.profiles.active 激活的配置
    • 使用 spring.profiles.include 激活的配置

  • 配置的优先级：激活的配置 > 默认配置

SpringBoot 的外部化配置记录

  • 官方文档

  • 当在代码中需要用到一些个属性是经常要变的，可以从 @Value 从配置文件中获取，而内部配置文件更改后要重新进行打包不方便，此时就要用到外部化配置了

  • 在 SpringBoot 中外部配置比内部配置的优先级高

  • 外部配置文件的方式：

    • 在 jar 包同目录下的配置文件 application.properties
    • 在 jar 包同目录下的 config 目录下的配置文件 application.properties
    • 在 jar 包同目录下的 config 目录下的直接目录下的配置文件 application.properties
    • 在 命令行以参数形式指定对应的属性

  • 外部配置文件方式的优先级：命令行方式 > config 目录下的目录下 > config 目录下 > jar 包同目录下 > jar 包内部配置

  • 当内部的激活配置和外部的默认配置同时生效时，使用的还是激活的配置，即激活配置优先级大于外部配置

  • 注解 @Test：被该注解标注则表示该方法是测试方法

  • 注解 @DisplayName：该注解作用是为测试类或测试方法自定义显示名称

  • 注解 @BeforeEach：表示注释的方法应在当前类中的每个 @Test、@RepeatedTest、@ParameterizedTest 或 @TestFactory 方法之前执行；

  • 注解 @AfterEach：表示注释的方法应在当前类中的每个 @Test、@RepeatedTest、@ParameterizedTest 或 @TestFactory 方法之后执行；

  • 注解 @BeforeAll：表示注释的方法应在当前类中的所有 @Test、@RepeatedTest、@ParameterizedTest 和 @TestFactory 方法之前执行；

  • 注解 @AfterAll：表示注释的方法应在当前类中的所有 @Test、@RepeatedTest、@ParameterizedTest 和 @TestFactory 方法之后执行；

SpringBoot 生命周期事件记录

  • 官方文档

  • 生命周期和事件关系简略图如图：

  • 系统九大事件：下面介绍由AI进行翻译

    • ApplicationStartingEvent：在应用启动时发送，但在任何处理之前，除了监听器和初始化器的注册。
    • ApplicationEnvironmentPreparedEvent：当环境已经确定，但在创建应用上下文之前发送。
    • ApplicationContextInitializedEvent：当应用上下文准备好并且应用上下文初始化器已被调用，但在加载任何 Bean 定义之前发送。
    • ApplicationPreparedEvent：在刷新操作开始之前发送，但在加载 Bean 定义之后。
    • ApplicationStartedEvent：在应用上下文已被刷新之后发送，但在调用任何应用和命令行运行器之前。
    • AvailabilityChangeEvent：紧接着发送，表示应用已被认为是活跃的，LivenessState 为 CORRECT。
    • ApplicationReadyEvent：在调用任何应用和命令行运行器之后发送。
    • AvailabilityChangeEvent：紧接着发送，表示应用已准备好接受请求，ReadinessState 为 ACCEPTING_TRAFFIC。
    • ApplicationFailedEvent：如果启动时发生异常，则发送此事件。
